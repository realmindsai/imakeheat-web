# SP-vibe character on the 12-bit button — Implementation Plan

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make the 12-bit button on the effects rack sound like an SP-1200 (drums hit harder, gentle grit) without altering 2/4/8/16, the pitch path, or any export format.

**Architecture:** Two surgical changes wired through TDD. (1) `bitcrusher.worklet.ts` gains a single conditional branch that pushes input through a `tanh` soft-clipper before quantizing — but only when `bits === 12`. (2) `session.ts` gains an `srManuallyAdjusted` boolean and a `nudgeSampleRate(hz)` action that sets sample rate without tripping that flag; `EffectsRack.tsx` calls the nudge once when 12-bit is selected for the first time and the user hasn't moved the SR slider. Spec: `docs/superpowers/specs/2026-04-30-sp-vibe-12bit-design.md`.

**Tech Stack:** TypeScript, React, Zustand, Vite, AudioWorklet, Vitest (jsdom), Playwright.

---

## File Structure

**Modify:**
- `src/audio/worklets/bitcrusher.worklet.ts` — add 12-bit-only `tanh` saturator before existing quantizer math.
- `src/store/session.ts` — add `srManuallyAdjusted` field + `nudgeSampleRate` action; track flag inside `setEffect` and `setSource`.
- `src/screens/EffectsRack.tsx` — when bit-depth changes from non-12 to 12, call `nudgeSampleRate(26000)` if flag is false.
- `tests/unit/bitcrusher.test.ts` — extend existing suite with saturator regression tests.
- `tests/unit/session-store.test.ts` — extend existing suite with flag/nudge tests.

**Create:**
- `tests/e2e/sp-vibe-nudge.spec.ts` — two e2e cases: fresh-session 12-tap nudges SR; manual SR move blocks future nudges.

**Why no separate file for the saturator:** It's ~3 lines of arithmetic. A separate module would be over-decomposition; the worklet is already the right home for "things that happen to the bit-crushed signal."

---

## Constants and conventions used in this plan

- **Drive:** +2 dB pre-saturation gain. As a linear factor this is `10 ** (2 / 20) ≈ 1.2589`. Define as `SP_DRIVE = 1.2589254117941673` at module scope in the worklet.
- **SP target rate:** `26000` Hz (round number — spec rationale: 26.04 kHz to four decimals is inaudible).
- **Soft-clip curve:** `Math.tanh(x * SP_DRIVE)`. Note: at typical signal levels `tanh(1.26 * x)` already provides graceful clamping in `[-1, 1]`.

### Task 3: Implement the 12-bit saturator

**Files:**
- Modify: `src/audio/worklets/bitcrusher.worklet.ts`

- [ ] **Step 1: Add the saturator branch**

Replace the body of the `process` method in `src/audio/worklets/bitcrusher.worklet.ts` so it reads:

```ts
import './processor-shim'

// SP-1200 vibe: pre-quantize tanh soft-clip on the 12-bit path only.
// Drive sits BEFORE the quantizer so the nonlinearity shapes what gets
// rounded — that's the bit that gives drums the "hit" people associate
// with the SP. Other bit depths are untouched.
const SP_DRIVE = 1.2589254117941673 // +2 dB

export class BitCrusherProcessor extends AudioWorkletProcessor {
  private bits = 16

  constructor() {
    super()
    ;(this as any).port.onmessage = (ev: MessageEvent) => {
      const data = ev.data as { bits?: number }
      if (typeof data.bits === 'number') {
        this.bits = Math.max(2, Math.min(16, Math.round(data.bits)))
      }
    }
  }

  process(
    inputs: Float32Array[][],
    outputs: Float32Array[][],
    _parameters: Record<string, Float32Array>,
  ): boolean {
    const input = inputs[0]
    const output = outputs[0]
    if (!input || input.length === 0) return true

    if (this.bits >= 16) {
      for (let c = 0; c < input.length; c++) {
        if (output[c]) output[c].set(input[c])
      }
      return true
    }

    const levels = 1 << this.bits      // 2^bits
    const half = levels / 2
    const spVibe = this.bits === 12

    for (let c = 0; c < input.length; c++) {
      const inCh = input[c]
      const outCh = output[c]
      if (!outCh) continue
      for (let i = 0; i < inCh.length; i++) {
        const x = spVibe ? Math.tanh(inCh[i] * SP_DRIVE) : inCh[i]
        const q = Math.max(-half, Math.min(half - 1, Math.round(x * half)))
        outCh[i] = q / half
      }
    }
    return true
  }
}

registerProcessor('bitcrusher', BitCrusherProcessor)
```

What changed: a module-scope `SP_DRIVE` constant, a `spVibe` boolean computed once per `process()` call, and a one-line ternary inside the inner loop. Everything else is byte-for-byte the original.

import { test, expect } from '@playwright/test'

function makeSine(freq: number, amp: number, durSec: number, sr: number): number[] {
  const n = Math.floor(durSec * sr)
  const out = new Array<number>(n)
  for (let i = 0; i < n; i++) out[i] = amp * Math.sin((2 * Math.PI * freq * i) / sr)
  return out
}

function rms(xs: number[]): number {
  let s = 0
  for (const v of xs) s += v * v
  return Math.sqrt(s / xs.length)
}

test('offline render at bitDepth=12 boosts a small-signal sine relative to bitDepth=16', async ({ page }) => {
  await page.goto('/integration-runner/')
  await page.waitForFunction(() => document.getElementById('status')?.textContent === 'ready')

  // -6 dBFS sine, far below the 12-bit lattice noise floor.
  const sr = 48000
  const sig = makeSine(1000, 0.5, 0.5, sr)
  const inRms = rms(sig)

  const baseEffects = {
    bitDepth: 16 as const,
    sampleRateHz: 48000,
    pitchSemitones: 0,
    speed: 1,
    filterValue: 0,
  }

  const result16 = await page.evaluate(async ({ pcm, sr, effects }) => {
    return await window.__run({
      kind: 'render',
      sourcePcm: [pcm],
      sampleRate: sr,
      effects,
      trim: { startSec: 0, endSec: pcm.length / sr },
    })
  }, { pcm: sig, sr, effects: baseEffects })

  const result12 = await page.evaluate(async ({ pcm, sr, effects }) => {
    return await window.__run({
      kind: 'render',
      sourcePcm: [pcm],
      sampleRate: sr,
      effects: { ...effects, bitDepth: 12 },
      trim: { startSec: 0, endSec: pcm.length / sr },
    })
  }, { pcm: sig, sr, effects: baseEffects })

  const out16Rms = rms(result16.pcm[0])
  const out12Rms = rms(result12.pcm[0])

  // 16-bit path is essentially identity; RMS ~= input.
  expect(Math.abs(out16Rms / inRms - 1)).toBeLessThan(0.05)

  // 12-bit path: +2 dB pre-quantize drive → RMS ratio noticeably above 1.
  // WSOLA + offline rendering introduces some envelope smearing, so we use a
  // looser tolerance than the unit test.
  expect(out12Rms / inRms).toBeGreaterThan(1.10)
  expect(out12Rms / inRms).toBeLessThan(1.40)
})

## Notes for the implementer

- **Don't reach for new abstractions.** The whole feature is one `if`, one boolean, one method, and one `onClick` branch. Each step has a complete, paste-ready code block — use them.
- **Don't change 2/4/8/16 paths.** The pre-existing tests and the new 8-bit-hard-clip regression test will catch a stray edit.

- **The `'sampleRateHz' in patch` check is intentional.** It distinguishes "user changed SR" from "user changed something else and SR is incidentally in the same store." `Partial<EffectParams>` does not include the key when the caller didn't pass it.
- **Why no React-component unit test for `EffectsRack`:** The project pattern uses Playwright e2e for screen-level wiring. The two e2e tests in Task 7 cover both the nudge-fires and nudge-suppressed paths.
